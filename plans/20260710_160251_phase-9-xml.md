# Plan — Phase 9: XML

**Status:** completed

Plan for building Phase 9 from [implementation-plan.md](../docs/implementation-plan.md)
(lines 569–623). This phase adds the **XML** format module: multiple views (pretty /
tree / raw), tree navigation, search + optional XPath, format handling + entities
(well-formedness, namespaces, comments/CDATA, entity round-trip), editing with a pre-save
well-formedness gate, plus insights / output / file-level ops (export, print, zip, copy,
split/merge).

It mirrors the JSON (Phase 8), CSV (Phase 7), Markdown (Phase 6), and TXT (Phase 4)
modules and reuses every Phase 3 (editor core) and Phase 5 (share / zip / print / export /
TTS) service unchanged. Phase 9 depends only on Phases 3 and 5 (both ✅ done); Phases 6–9
are independent, so building XML now is in order.

---

## 1. What this phase is

Phase 9 has six tasks:

- **9.1 Views: pretty / tree / raw** — a syntax-highlighted, indented pretty view; a
  collapsible element tree; the raw source as-is; toggle between them; remember position.
- **9.2 Tree navigation** — expand/collapse one or all; tag name, attributes, text value;
  child counts and repeated elements; copy a node's path (`root/items/item[2]/name`); copy
  a subtree / an attribute / a value.
- **9.3 Search + optional XPath** — search by tag / attribute / text, jump between matches,
  filter the tree; an optional XPath query; line numbers + tag/bracket matching in raw.
- **9.4 Format handling + entities** — well-formedness check (error line if not);
  format / minify; handle the XML declaration / encoding; show namespaces; collapse
  comments and CDATA; resolve built-in and numeric entities for display and re-escape them
  correctly on save so text is never corrupted; **(advanced) XSD validation** via a
  platform channel. DTD and XSLT are out of scope.
- **9.5 Editing with a pre-save gate** — edit element text / attributes / structure in the
  tree and the raw source; add / delete / move elements and attributes; undo/redo, find &
  replace; a well-formedness check **blocks an invalid save** (or save-as-a-copy); preserve
  the XML declaration and encoding on save (or a chosen encoding / line ending).
- **9.6 Insights, output, file-level ops** — element count, depth, most-common tags,
  attribute list per element type; export XML → JSON / CSV (flatten a repeated element) /
  PDF / HTML; print; zip; copy full / minified / subtree; metadata (root element, encoding,
  element count); split by a repeated child / merge under a new wrapper root.

The module follows the established pattern: **pure-Dart logic classes** (parser wrapper,
stats, path/XPath, split/merge, tree edits, convert) unit-tested with no device, wrapped by
**thin Riverpod/widget UI** that only wires providers and gathers input.

## 2. What the issue is

The critical path is `… → 5 → (6|7|8|9) → 10 …`. Phase 5 is done and proven on TXT, MD,
CSV and JSON, but XML files still fall through to the placeholder view:
`format_dispatch.dart` already returns `DocumentFormat.xml` (by `.xml` extension or
`application/xml` / `text/xml` MIME), yet `tabs_workspace.dart` only special-cases txt /
markdown / json / csv, so a `.xml` file opens as an inert placeholder. `lib/formats/xml/`
is an empty placeholder folder. Phase 9 fills it and wires it into the shell.

## 3. Key design decisions

### 3.1 Use the `xml` package (the one new dependency this phase introduces)

The implementation plan's *Introduces (packages)* line for Phase 9 names **`xml`**. Unlike
JSON (where `dart:convert` forced a custom reader), the `xml` package (Lukas Renggli,
**MIT** — open source, CLAUDE.md §3.1) is the right tool for XML: it parses to a DOM with
elements, attributes, text, comments, CDATA, processing instructions, the XML declaration,
and namespaces; it **decodes built-in + numeric entities** on read and **re-escapes** them
on serialize (task 9.4's entity round-trip requirement); and it serializes with a pretty
printer and a minifier. We will pin a recent 6.x (`xml: ^6.5.0`).

`package:xml/xpath.dart` (part of the same package — **no extra dependency**) gives a
pragmatic XPath for task 9.3. Its subset and limits are documented; an unsupported /
invalid query degrades to a friendly "no matches / invalid query" message, never a crash.

### 3.2 A never-throwing parser wrapper

`xml_parser.dart` wraps `XmlDocument.parse` in an `XmlParseResult` (`ok`, `document`,
`errorMessage`, `errorLine`, `errorColumn`). `XmlParserException` from the package already
carries a 1-based line/column, which we surface for the validity chip and the pre-save gate
message. The wrapper **never throws** on bad input (CLAUDE.md §3.4). It also exposes
`pretty(doc, indent)` (via `toXmlString(pretty: true, indent: …)`), `minify(doc)` (via
`toXmlString()` with no whitespace), the declaration/encoding, and the namespace list.

### 3.3 Colour-coded pretty view without extra packages

Like the JSON pretty view, the XML pretty view is coloured rich text
(`SelectableText.rich`) built by walking the DOM — tag names, attribute names, attribute
values, text, comments, and CDATA each get a theme-aware colour, so it works in
light/dark/sepia with **no new rendering dependency**. The raw surface uses `re_editor`
(line-number gutter, find panel, bracket matching), exactly as the other formats use it.

### 3.4 Tree editing via DOM-mutate-then-re-serialize (design difference from JSON)

JSON tracked source spans so a tree edit was a precise text-span replacement. The `xml`
package's DOM does **not** expose per-node source offsets, and building a span-tracking XML
parser would duplicate the package. So for XML, a **structural tree edit** is applied by
mutating a parsed copy of the document and **re-serializing** (pretty-printed with the
chosen indent), then setting the editor buffer via the session's `applySource` — which
flows back through `re_editor`, so **undo/redo, dirty tracking, and the save pipeline keep
working** unchanged.

Trade-off (documented): re-serializing normalizes whitespace across the document, so a tree
edit reformats the file. This is consistent with the format/minify action being available
anyway, and the **raw editor remains the precise, free-form editing surface** for users who
want to preserve exact byte layout. The XML **declaration is preserved** because the
package keeps the `XmlDeclaration` node and re-emits it. First cut covers edit-text,
edit/add/delete-attribute, rename-element, add-child, delete-node, and move-sibling
(up/down); drag-reorder is later polish and blocks no 9.x acceptance.

### 3.5 The pre-save well-formedness gate (9.5)

`XmlWellFormedGate implements SaveGate` (the seam Phase 3 built into `AtomicSaver`).
`save()` passes the gate, so an invalid overwrite is **blocked** with the parse error line;
`saveAsCopy()` does **not** run the gate (the built-in escape hatch) — exactly the plan's
"or save-as-a-copy". A **plain save writes the raw buffer bytes** through `AtomicSaver` +
`SafSaveTarget`, so the declaration, encoding, and formatting the user typed are preserved
by default (task 9.5); the chosen encoding / line ending are honoured.

### 3.6 XSD validation — **deferred** (decision to confirm)

Task 9.4 marks XSD validation **"(advanced)"** and optional, to be done "via a platform
channel" using Android `javax.xml.validation`. That needs new Kotlin native code plus its
own manual on-device verification. To keep this phase shippable and its automated tests
device-free (the project's convention), I propose to **deliver well-formedness validation
now and defer full XSD to a follow-up**, recorded as **owed** in the progress notes — the
same way earlier phases deferred native-only work (e.g. the SAF on-device checks). The
Validate sheet will clearly show "Well-formed ✓/✗ (line N)" and note that XSD schema
validation is coming. **DTD and XSLT stay out of scope** per the plan.

If you would rather I build the XSD platform channel now as part of Phase 9, say so and I
will add the Kotlin `javax.xml.validation` channel + a Dart wrapper + its owed manual check
to the plan.

### 3.7 Export targets — no enum change needed

XML exports to **JSON / CSV / PDF / HTML / plain text** (task 9.6). `ExportTarget` already
has `json`, `csv`, `pdf`, `html`, and `plainText` (added in Phases 7–8), so **no change to
`export_target.dart`**. XML→JSON maps elements to objects (attributes as `@attr`, text as
`#text`, repeated tags to arrays); XML→CSV flattens a chosen repeated child element into
rows (union of child-tag/attribute names as the header); XML→PDF/HTML reuse the shared
`PdfWriter`/`HtmlWriter` over the pretty XML. There is no `xml` `ExportTarget` (self-export
is just the file itself), matching the plan's list.

### 3.8 Packages & permissions

**One new package: `xml` (MIT).** No new Android permissions; remote content is not fetched
(offline-first, CLAUDE.md §3.2). XPath uses `package:xml/xpath.dart` (same package).

## 4. Files to be changed / created

Module folder: **`lib/formats/xml/`**, file prefix **`xml_`** (mirrors `json_` / `csv_`).

### New — pure Dart logic (no Flutter; directly unit-tested)

- `lib/formats/xml/xml_parser.dart` — the never-throwing wrapper over `XmlDocument.parse`
  (§3.2): `XmlParseResult` (`ok`, `document`, `errorMessage`, `errorLine`, `errorColumn`);
  `pretty(doc, indent)`, `minify(doc)`; declaration/encoding readout; namespace list.
- `lib/formats/xml/xml_path.dart` — `pathOf(node)` → `root/items/item[2]/name` (index only
  when a tag repeats); an XPath query via `package:xml/xpath.dart` returning matching nodes;
  bad/unsupported queries → empty + friendly message, never throw.
- `lib/formats/xml/xml_stats.dart` — `XmlStats`: element count, max depth, most-common tags,
  attribute names per element type, root element name, declared encoding.
- `lib/formats/xml/xml_split_merge.dart` — split a document into N parts by a chosen
  **repeated child element** (each part = the root wrapper + a slice of those children); and
  merge several documents' root children **under a new wrapper root**. Round-trip friendly.
- `lib/formats/xml/xml_tree_edits.dart` — pure DOM edit operations (§3.4) that take a source
  string + a node locator and return the new serialized source: `setText`, `setAttribute`,
  `removeAttribute`, `renameElement`, `addChild`, `deleteNode`, `moveSibling`.
- `lib/formats/xml/xml_convert.dart` — pure `xmlToJson(document)` and
  `xmlToCsv(document, repeatedTag)` used by the exporter (§3.7).
- `lib/formats/xml/xml_well_formed_gate.dart` — `XmlWellFormedGate implements SaveGate`
  (§3.5): parses; returns `GateResult.invalid(message-with-line)` on failure.

### New — views (Flutter widgets)

- `lib/formats/xml/xml_pretty_view.dart` — colour-coded, indented `SelectableText.rich`
  pretty view built from the DOM (§3.3); scroll position remembered by fingerprint.
- `lib/formats/xml/xml_tree_view.dart` — the collapsible element tree: expand/collapse one
  or all, tag name + attribute chips + text preview, child-count / repeated-element labels,
  tap actions (copy path / value / attribute / subtree; in edit mode: edit text, edit/add/
  delete attribute, rename, add child, delete, move up/down), filter-to-matches for search.
- `lib/formats/xml/xml_editor_surface.dart` — the raw source surface via `re_editor`
  (gutter, find panel, bracket matching). Editable.
- `lib/formats/xml/xml_find_panel.dart` — tag/attribute/text search controls + an XPath
  query box with a friendly "no matches / invalid query" state.
- `lib/formats/xml/xml_read_aloud_button.dart` — read-aloud button (mirrors the JSON one).

### New — state & shell glue (mirror JSON/CSV)

- `lib/formats/xml/xml_document_session.dart` — `XmlDocumentSession extends ChangeNotifier`
  (no Riverpod). Loads bytes via SAF, decodes (never throws), holds the `re_editor`
  controllers for the source, tracks a **view mode** (`pretty` / `tree` / `raw` / `edit`),
  parses to a DOM + validity + stats + namespace/declaration readout, drives drafts +
  autosave, applies tree edits via `applySource`, and saves via `AtomicSaver` +
  `SafSaveTarget` **with `XmlWellFormedGate`**, preserving encoding/line endings (chosen
  indent for format/minify). Builds metadata with `formatFields` (root element, encoding,
  element count, depth). Mirrors `JsonDocumentSession`.
- `lib/formats/xml/xml_session_manager.dart` — `XmlSessionManager` +
  `xmlSessionManagerProvider` (`sessionFor` / `peek` / `release` / `retainOnly`).
- `lib/formats/xml/xml_document_view.dart` — the tab body: loading spinner / friendly
  failure view (Retry) / ready view; ready view shows the draft banner + the pretty / tree /
  raw / edit view for the current mode, with a tree search box above the tree.
- `lib/formats/xml/xml_toolbar.dart` — action bar: view toggles (pretty / tree / raw /
  edit), undo/redo (edit), find, format / minify, validate, expand/collapse all (tree),
  save, read-aloud, and an overflow menu (find & replace, XPath, insights, file info, split,
  merge, copy full/minified, share, share-as-zip, print, export). Mirrors `JsonToolbar`.

### New — output helpers (mirror JSON/CSV, reuse Phase 5 services)

- `lib/formats/xml/xml_exporter.dart` — `XmlExporter implements FormatExporter`,
  `formatId => 'xml'`, targets `{json, csv, pdf, html, plainText}`. JSON via
  `xmlToJson`, CSV via `xmlToCsv`, PDF/HTML via the shared writers over pretty XML,
  plainText passes the source through. Unsupported target → `UnsupportedExportException`.
  Lives in the XML module (needs the parser), registered via DI — same choice as
  `JsonExporter`.
- `lib/formats/xml/xml_output_actions.dart` — `XmlOutputActions` (`formatId = 'xml'`):
  `shareFile`, `shareAsZip`, `printDoc`, `runExport`, `saveExport`, `shareExport`.
- `lib/formats/xml/xml_export_sheet.dart` — target picker → run → share/save (CSV target
  asks which repeated element to flatten).
- `lib/formats/xml/xml_info_sheet.dart` — stats + metadata + insights rows.
- `lib/formats/xml/xml_save_options_sheet.dart` — indent (2 / 4 / tab) + encoding +
  line-ending choose → save.
- `lib/formats/xml/xml_split_merge_actions.dart` — split dialog (pick the repeated child) →
  per-part `saf.createDocument`; merge picks XML files and wraps them under a new root.
- `lib/formats/xml/xml_tools_sheets.dart` — the XPath-query and validate sheets (validate
  shows well-formedness + the XSD-deferred note, §3.6). Mirrors `json_tools_sheets.dart`.

### Changed — wiring

- `lib/shell/tabs/tabs_workspace.dart` — add a `DocumentFormat.xml` branch in the **three**

- `lib/formats/format_dispatch.dart` — (minor) accept a few extra XML extensions
  (`xsd`, `xsl`, `svg`, `xhtml`, `rss`, `atom`) so those open in the XML viewer. No

### New — tests (`test/formats/xml/`, mirroring `test/formats/json/`)

- `xml_stats_test.dart` — element count / depth / most-common tags / attributes per element.
- `xml_split_merge_test.dart` — **split by a repeated element then merge under a wrapper

- `xml_convert_test.dart` — `xmlToJson` and `xmlToCsv` produce the expected structure.
- `xml_exporter_test.dart` — XML→JSON is valid JSON; XML→CSV flattens a repeated element;

## 6. How we honour the project rules

- **Open source only** — the one new package (`xml`) is MIT; no Syncfusion (CLAUDE.md §3.1).
- **Never crash on bad input** — the parser wrapper never throws; corrupt/empty/wrong-
  encoding lands in a friendly `failed` state; invalid XPath degrades to a message
  (CLAUDE.md §3.4).
- **Scoped storage only** — open / save / merge / split all go through SAF +
  `createDocument`; no broad storage (CLAUDE.md §3.3).
- **Atomic saves + preserve encoding/line endings** — reuse `AtomicSaver` + `SafSaveTarget`
  exactly as JSON/CSV, now with the XML well-formedness gate; the raw buffer save preserves
  the declaration/encoding (CLAUDE.md §3.5).
- **Never lose edits silently** — reuse drafts/autosave + the shell's unsaved-changes close
  guard; the gate blocks a broken overwrite but offers save-as-a-copy (CLAUDE.md §3.6).
- **Offline-first** — no remote fetch; exports are self-contained (CLAUDE.md §3.2).
- **Simple English** — all UI strings, the change log, and comments in plain English.

## 8. Out of scope for this phase (noted, not dropped)

- **XSD validation** — deferred to a platform-channel follow-up (§3.6); well-formedness ships
  now. **DTD and XSLT** stay out of scope per the plan.

  unsupported queries degrade gracefully.
- **Advanced structural tree edits** (drag-reorder, arbitrary move) — first cut is
  edit-text / attributes / rename / add-child / delete / move-sibling (§3.4).

- **Syntax colouring inside the raw `re_editor` surface** — the *pretty* view is coloured;
  the raw editor is plain monospace (a later polish item, matching JSON/MD).
