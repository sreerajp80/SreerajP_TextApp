# Fix recent files not opening in the editor

**Status:** completed

## Issue

Tapping an available item on the Recent files screen reads the file, but the
screen refreshes and remains on Recent files instead of opening the editor.

The tap callback is built by `_RecentsList`, and `OpenFileAction` keeps using
that list widget's `WidgetRef` after an asynchronous call. `recordOpen()`
refreshes `recentsControllerProvider`; its loading state removes `_RecentsList`
from the widget tree and disposes that ref. The later calls that should add the
tab and select the Editor destination therefore use a disposed ref and do not
complete.

## Files to change

- `lib/shell/open_file_action.dart` — make the open flow safe across async
  provider refreshes by obtaining the required service and controllers before
  the first asynchronous gap, then use those saved references to record the
  recent, open/focus the tab, and select Editor.
- `test/shell/home_screen_widget_test.dart` — add a regression test that makes
  `recordOpen()` refresh and dispose the recent list, then verifies that tapping
  the item still opens a tab and changes the shell destination to Editor.
- `test/support/test_support.dart` — extend the test recents controller with the
  small refresh behavior needed to reproduce the disposed-ref failure.
- `change_log/<timestamp>_recent-file-open.md` — record the implemented change
  and test result after the fix is complete.

## Plan for the fix

1. Capture the SAF service, recents controller, tabs controller, and shell
   destination controller at the start of the open operation, while the tap
   widget's ref is valid.
2. Keep the current SAF error handling, fingerprinting, tab-cap behavior, and
   Recent files refresh behavior unchanged.
3. Use the captured controllers after asynchronous work so rebuilding or
   disposing the tapped list row cannot interrupt opening the editor.
4. Add a widget regression test that reproduces the refresh during a recent
   file tap and checks both the opened tab and Editor destination.
5. Run formatting, the focused widget test, static analysis, and the relevant
   shell test suite. Then create the required change log and mark this plan
   completed.

## Security and data safety

- The fix keeps scoped-storage access through the existing SAF service.
- File bytes remain local and are used only for the existing fingerprint.
- No file contents, URI grants, save behavior, or unsaved-edit rules change.
