# Plan — Phase 4: TXT format (first vertical slice)

**Status:** completed

Implements Phase 4 of [implementation-plan.md](../docs/implementation-plan.md)
(tasks 4.1–4.5). Goal: prove the whole **open → view → edit → save** slice on the
simplest format (plain text) before scaling to Markdown, CSV, JSON, and XML.

---

## 1. What the issue is

Phases 0–3 built the foundation, the app shell with tabs, and the shared editor
core — but **all of it is pure services with no on-screen editor yet**. Today an
open tab shows [placeholder_document_view.dart](../lib/shell/tabs/placeholder_document_view.dart),
which only prints the file name and size. Nothing loads the file's text, shows it,
edits it, or saves it. The tab close guard even says *"Saving becomes available with
the editor."*

Phase 4 fills that gap for **TXT files only**: load the bytes, decode them, show the
text with a real viewer and editor, and save changes back through the atomic saver —
wiring together the Phase 3 services (`TextCodecService`, `EditorController`,
`TextSearch`/`FindReplace`, `AtomicSaver` + `SafSaveTarget`, `DraftStore`/`AutoSaver`,
`MetadataService`) into the first working format.

## 2. Decisions taken before this plan

Two choices were confirmed with the user (they change the plan):

1. **Editor surface = an open-source editor package.** We add **`re_editor`**
   (MIT license — open-source, allowed under CLAUDE.md §3.1; actively maintained).
   It provides the text/code surface with a **line-number gutter**, undo/redo,
   read-only mode, and a find/replace control model out of the box. This is the
   surface for TXT editing.
   - **Design tension noted:** `re_editor` brings its **own** editing controller
     (`CodeLineEditingController`) with its own undo/redo and find/replace. For TXT
     we let `re_editor` own the live editing surface (text + undo/redo + gutter +
     find UI). The Phase 3 pure-Dart `EditorController`/`FindReplace`/`UndoRedoStack`
     stay in the codebase and remain the model for the **structured** formats
     (JSON/XML tree edits, CSV grid) where the surface is not a plain text box. We do
     **not** try to force `re_editor` through `EditorController`; instead the TXT
     session reads text out of `re_editor`'s controller for save/stats/draft. This
     keeps both simple and avoids a fragile double-controller bridge.
   - The shared **load/decode, atomic save, draft/auto-save, metadata, encoding,
     stats, and split/merge** services are all still used — only the *typing surface*
     comes from the package.

2. **Clickable URLs = add `url_launcher`, gated by a warning.** Task 4.1 requires
   clickable URLs. `url_launcher` (BSD, Flutter-team) is the standard open-source way
   to open a link. It is also needed again in **Phase 6** (Markdown autolinks) and by
   the "open remote links" optional-online feature (CLAUDE.md §3.2), so adding it now
   is not throwaway. This is a small, deliberate deviation from the plan's "no new
   packages for Phase 4" note.
   - **Tapping a URL never opens the browser straight away.** It first shows a
     **warning dialog** — the full URL as selectable text plus a note that opening a
     link leaves the app and goes online. The dialog offers **Open in browser**
     (launches the default browser via `url_launcher` only on explicit accept),
     **Copy link** (always available), and **Cancel** (default). If the user cancels,
     nothing is launched. This respects the offline-first stance and the security
     rule about treating opened-file content as untrusted (a link inside a file could
     point anywhere).

Both packages are MIT/BSD open-source; neither is Syncfusion or source-available.

## 4. Design overview

### 4.1 The per-tab TXT "document session"

The core new idea is a **document session**: a per-tab object that owns everything
about one open TXT file. It is the bridge between a shell `DocumentTab` (which only
holds URI + fingerprint + name) and the on-screen viewer/editor.

A `TxtDocumentController` (Riverpod `AutoDisposeNotifier` **family**, keyed by tab id)
holds a sealed state:

- `TxtLoading` — reading bytes.
- `TxtReady` — decoded text is in hand; holds the `re_editor` controller, the detected
  `TextEncodingType` + `LineEndingStyle`, `isWritable`, `FileMetadata`, live stats, the
  view mode (view/edit), word-wrap flag, and a pending-draft flag.
- `TxtFailed` — a friendly message (read error / looks-like-binary), never a crash.

Responsibilities:

- **Load** bytes via `SafService.readBytes`, decode via `TextCodecService`
  (never throws), and sniff for "not text" (many NUL / replacement chars) → offer a
  friendly notice with "view raw anyway".
- **Draft restore:** on load, if `DraftStore.hasDraft(fingerprint)`, surface a
  restore/discard choice; a real save clears the draft.
- **Auto-save:** run an `AutoSaver` on the editor content while editing.
- **Dirty tracking:** push the editor's dirty flag to
  `TabsController.setDirty(tabId, …)` so existing close guards work.
- **Save / Save as a copy:** through `AtomicSaver` + `SafSaveTarget`, honoring the
  chosen encoding + line ending, then `markSaved` + clear draft.
- **Reading position:** persist the first-visible line index keyed to the fingerprint
  in `KeyValueStore`, and restore it on open (survives font/wrap changes because it is
  a line index, not a pixel offset).

### 4.2 Providers (shared editor DI)

New `lib/core/editor/editor_providers.dart` exposing the Phase 3 services that had no
provider yet: `textCodecServiceProvider`, `atomicSaverProvider`,
`metadataServiceProvider`, plus a temp-dir provider for `SafSaveTarget`. Reused by the
session and by later formats.

### 4.3 Wiring into the shell

- A tiny **format dispatch** (`lib/formats/format_dispatch.dart`): given a tab's
  mime/extension, return the right document body. TXT (and unknown-but-texty) →
  `TxtDocumentView`; everything else → the existing placeholder (until its phase).
- [tabs_workspace.dart](../lib/shell/tabs/tabs_workspace.dart):
  - Replace the direct `PlaceholderDocumentView` with the dispatcher.
  - Turn `_DocumentToolbar` into the real TXT toolbar host (view/edit toggle, undo/redo,
    find & replace, save menu, word-wrap, jump-to-line, encoding, info/stats,
    split/merge, keeping the existing read-only lock). Actions come from the active
    session; disabled while loading/failed.
  - Replace the "Saving becomes available with the editor" stub in `_confirmClose`
    with a real Save / Save as a copy path driven by the session.

### 4.4 Viewer vs editor

- **Viewer** (`TxtViewer`): read mode. `re_editor` in read-only mode gives the
  line-number gutter + scroll; word-wrap toggle; jump-to-line; **clickable URLs**
  (detect URLs; tapping opens the warning dialog described in §2.2 → Open in browser /
  Copy link / Cancel); honors font size/family/line-spacing from `ThemeSettings` and
  the light/dark/sepia theme.
- **Editor** (`TxtEditor`): edit mode. `re_editor` editable, with undo/redo and
  find & replace; the toolbar's Save / Save-as-a-copy opens an encoding + line-ending
  chooser (`TxtSaveOptionsSheet`).

### 4.5 Pure logic (easy to unit-test)

- `TxtStats.of(text)` → word / character / line counts (handles empty + multi-line).
- `TxtSplitMerge`:
  - `splitByLines(text, linesPerPart)` and `splitBySize(text, maxBytes, encoding)`.
  - `merge(List<String> parts)` = ordered concatenation.
  - Round-trip property: `merge(splitByLines(t)) == t`.
- Binary/"not text" sniff helper (used by the failure path).

## 7. Rules kept in view (CLAUDE.md)

- **§3.4 never crash on bad input:** decode never throws; load failures and
  binary-looking files show a friendly state; regex find errors already handled by
  `TextSearch`.
- **§3.5 atomic saves:** saving goes only through `AtomicSaver` + `SafSaveTarget`
  (temp-write-verify then replace); encoding + line endings preserved by default.
- **§3.6 never lose edits silently:** draft/auto-save restore on open; dirty flag feeds
  the existing unsaved-changes close prompt; read-only lock respected.
- **§3.3 scoped storage only:** all file access stays on `SafService`; no new
  permissions, no file browser.
- **§3.1 open source only:** `re_editor` (MIT), `url_launcher` (BSD) — both verified.

## 8. How we know Phase 4 is done

- `flutter analyze` clean; `flutter test` green including the six new test files.
- A TXT file opens into a real viewer, toggles wrap, jumps to a line, restores its
  reading position, and opens links.
- Editing with undo/redo + find & replace, then Save / Save as a copy round-trips
  through the atomic saver with encoding + line endings preserved; read-only offers
  copy only.
- Stats + metadata show; encoding can be viewed/switched; corrupt/empty/truncated/
  wrong-encoding files show a friendly message, never a crash.
- Split then merge reproduces the original.
- One manual on-device pass (owed, recorded in the change log): open a real TXT via
  SAF, edit, overwrite-save, save-as-copy, switch encoding, open a link.
