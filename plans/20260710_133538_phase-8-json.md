# Plan — Phase 8: JSON

**Status:** completed

Plan for building Phase 8 from [implementation-plan.md](../docs/implementation-plan.md)
(lines 515–565). This phase adds the **JSON** format module: multiple views (pretty /
tree / raw / minified), tree navigation, search + optional JSONPath, validation +
variants (JSONC/JSON5/NDJSON) + big-number safety, editing with a pre-save
well-formedness gate, plus insights / output / file-level ops (export, diff, split/merge).
It mirrors the TXT (Phase 4) and Markdown (Phase 6) modules and reuses every Phase 3
(editor core) and Phase 5 (share / zip / print / export / TTS) service unchanged.

Phase 8 depends only on Phases 3 and 5 (both ✅ done). Phase 7 (CSV) is **not** a
dependency — Phases 6–9 are independent — so building JSON now is in order.

---

## 1. What this phase is

Phase 8 has six tasks:

- **8.1 Views: pretty / tree / raw / minified** — an indented, colour-coded pretty view; a
  collapsible tree; the raw source as-is; a single-line minified view; toggle between them.
- **8.2 Tree navigation** — expand/collapse one or all; keys shown with value types; array
  indexes and child counts; copy a node's path (`data.users[3].name`); copy a subtree or a
  value.
- **8.3 Search + optional JSONPath** — search by key or value, jump between matches, filter
  the tree to matches; an optional JSONPath query; line numbers + bracket matching in the
  raw/pretty views.
- **8.4 Validation + variants + big numbers** — validate (well-formed, with the error line
  if not); optional JSON Schema validation listing errors; format / minify; NDJSON shown as
  a record list; lenient JSONC/JSON5 read (tolerates `//`, `/* */`, trailing commas) that
  **saves strict JSON** and tells the user; large / high-precision numbers kept as exact
  original text.
- **8.5 Editing with a pre-save gate** — edit values/keys in the tree and the raw source;
  add/delete nodes; undo/redo, find & replace; a well-formedness check runs before saving
  and **blocks an invalid save** (or save-anyway-as-a-copy); save with a chosen indentation
  and encoding/line ending.
- **8.6 Insights, output, file-level ops** — key count, depth, array sizes, type breakdown;
  export JSON → CSV (flat arrays) / YAML / PDF / HTML; diff two JSON files; print; zip; copy
  full / minified / subtree; metadata (top-level type, item count); split / merge top-level
  arrays.

The module follows the established pattern: **pure-Dart logic classes** (parser, stats,
path, schema, diff, split/merge, tree edits) that are unit-tested with no device, wrapped
by **thin Riverpod/widget UI** that only wires providers and gathers input. The genuinely
new pieces are a **custom lenient JSON reader** and a **collapsible tree widget** — see §3.

## 3. Key design decisions

### 3.1 A custom lenient JSON reader (the heart of the phase)

`dart:convert`'s `jsonDecode` cannot meet three of the plan's requirements at once:

1. **Exact big / high-precision numbers.** `jsonDecode` parses numbers into `int`/`double`,
   which **rounds** high-precision values and loses the original text. The plan requires
   large/high-precision numbers be **kept as exact original text** and survive a round-trip
   unrounded.
2. **Lenient JSONC/JSON5.** `jsonDecode` rejects `//` and `/* */` comments and trailing
   commas. The plan requires reading them, then saving **strict** JSON with a notice.
3. **Source spans for the tree and tree editing.** We need each node to know its
   `[start, end)` offset in the source so tree navigation, "copy subtree", and tree edits
   (8.5) can map a node to precise text.

So we build our own small tokenizer + parser (`json_parser.dart`) producing a `JsonNode`
tree (`json_node.dart`). This is exactly the "custom lenient reader" the plan's
*Introduces (packages)* line names ("Dart core `dart:convert`; a custom lenient reader for
JSONC/JSON5 and NDJSON"). It:

- Parses standard JSON strictly (used for the well-formedness gate and validation).
- In lenient mode tolerates `//` line and `/* */` block comments, trailing commas, and a
  reasonable **JSON5 subset** (single-quoted strings, unquoted identifier keys). Full JSON5
  (hex numbers, `+Infinity`, etc.) is a noted subset, not the whole spec.
- Keeps every number as its **raw source text** (a `JsonNumber` carries the original
  string), so nothing is rounded; `int.tryParse`/`double.tryParse` are used only for
  insights, never for round-tripping.
- Records each node's source span, its key, and child order (insertion order preserved).
- **Never throws** on bad input — it returns a result carrying `ok` + a friendly error
  message with a **1-based line/column** (CLAUDE.md §3.4).
- Provides `pretty(indent)` and `minify()` serializers that emit strict JSON from the tree,
  preserving raw number text.
- Detects and parses **NDJSON** (each non-blank line a JSON value) into a record list.

### 3.2 Colour-coded pretty view without a new package

The pretty view is rendered as coloured rich text (`SelectableText.rich`) built by walking
the `JsonNode` tree — keys, strings, numbers, booleans, and `null` each get a theme-aware
colour. This gives colour coding with **no new dependency** and full light/dark/sepia
control. The raw and minified surfaces use `re_editor` (line-number gutter, find panel,
bracket matching), exactly as TXT/MD use it.

### 3.3 Tree editing via source-span replacement

The **raw source text is the single source of truth** (held in the `re_editor`
controller). A tree edit (edit a scalar value, edit a key, delete a node, add a child to an
object/array) is applied as a **precise text-span replacement** using the node's recorded
`[start, end)` offset, then flows back through the editor — so undo/redo, dirty tracking,
and the save pipeline all keep working unchanged. First cut covers edit-value, edit-key,
delete, and add-child; more advanced structural moves (drag-reorder, move subtree) are
noted as later polish and do not block any 8.x acceptance.

### 3.4 The pre-save well-formedness gate (8.5)

We add a `JsonWellFormedGate implements SaveGate` (the seam Phase 3 built into
`AtomicSaver`). `save()` passes the gate, so an invalid overwrite is **blocked** with the
parse error; `saveAsCopy()` does **not** run the gate (that is the built-in AtomicSaver
behaviour), which is exactly the plan's "or save-anyway-as-a-copy" escape hatch. Today the
sessions call `_saver.save(...)` with no `gate:` argument; the JSON session adds it.

### 3.5 JSON Schema validation — a pragmatic subset

"Optional JSON Schema validation" is implemented as a **hand-written subset validator**
(`type`, `required`, `properties`, `items`, `enum`, `minimum`/`maximum`,
`minLength`/`maxLength`, `additionalProperties` bool) returning a list of `{path, message}`
errors. No new package (consistent with the project's "hand-roll small tolerant readers"
choice for MD front matter). The subset and its limits are documented; unknown keywords are
ignored rather than failing.

### 3.6 New export targets

`ExportTarget` currently has `pdf, docx, html, markdown, plainText`. JSON needs **`csv`**
and **`yaml`**; we add both to the enum and to its (exhaustive) `extension` / `mimeType` /
`label` switches. JSON→CSV flattens a top-level array of objects into rows (union of keys
as the header); JSON→YAML uses a small hand-written emitter; JSON→PDF/HTML reuse the shared
`PdfWriter`/`HtmlWriter` over the pretty text (HTML wraps the coloured/`<pre>` output).

### 3.7 Packages & permissions

**No new package** — `dart:convert` plus our custom reader cover parsing; YAML export is a
tiny hand emitter; colour coding is our own rich text. **No new Android permissions.**
Remote content is not fetched (offline-first, CLAUDE.md §3.2).

## 4. Files to be changed / created

Module folder: **`lib/formats/json/`**, file prefix **`json_`** (mirrors `txt_` / `md_`).

### New — pure Dart logic (no Flutter; directly unit-tested)

- `lib/formats/json/json_node.dart` — the `JsonNode` model: kind (object / array / string /
  number / bool / null), optional key, scalar value, ordered children, source span
  `[start, end)`, and raw number text. Immutable, with helpers (`isContainer`, `childCount`).
- `lib/formats/json/json_parser.dart` — the custom tokenizer + parser (§3.1). `JsonParseResult`
  (`ok`, `root`, `errorMessage`, `errorLine`, `errorColumn`); strict + lenient modes; comment
  / trailing-comma / JSON5-subset tolerance; raw-number preservation; source spans. Plus
  `pretty(node, indent)`, `minify(node)`, and `NdjsonReader` (records list). Never throws.
- `lib/formats/json/json_path.dart` — `pathOf(node)` → `data.users[3].name`; and a subset
  **JSONPath** evaluator (`$`, `.key`, `['key']`, `[index]`, `*`, recursive `..`) returning
  matching nodes. Bad queries return empty + a friendly message, never throw.
- `lib/formats/json/json_stats.dart` — `JsonStats`: key count, max depth, array sizes,
  type breakdown, top-level type, top-level item count.
- `lib/formats/json/json_schema_validator.dart` — the subset validator (§3.5) →
  `List<JsonSchemaError>`.
- `lib/formats/json/json_diff.dart` — `JsonDiff.compare(a, b)` → added / removed / changed
  path lists between two parsed trees.
- `lib/formats/json/json_split_merge.dart` — split a top-level array into N parts (by item
  count) and merge several top-level arrays by concatenation. Round-trip friendly.
- `lib/formats/json/json_tree_edits.dart` — pure text-span edit operations (§3.3):
  `setValue`, `setKey`, `deleteNode`, `addChild` — each takes the source + a node and
  returns the new source string.
- `lib/formats/json/json_yaml.dart` — a small YAML emitter for the export target.

### New — views (Flutter widgets)

- `lib/formats/json/json_pretty_view.dart` — colour-coded, indented `SelectableText.rich`
  pretty view built from the tree (§3.2); scroll position remembered by fingerprint.
- `lib/formats/json/json_tree_view.dart` — the collapsible tree widget: expand/collapse one
  or all, type badges, array index + child-count labels, tap actions (copy path / value /
  subtree; in edit mode: edit value/key, add, delete), filter-to-matches for search.
- `lib/formats/json/json_editor_surface.dart` — the raw / minified source surface via
  `re_editor` (gutter, word-wrap off by default, find panel, bracket matching). Editable.
- `lib/formats/json/json_find_panel.dart` — key/value search controls + a JSONPath query box
  with a friendly "no matches / invalid query" state.

### New — state & shell glue (mirror MD/TXT)

- `lib/formats/json/json_document_session.dart` — `JsonDocumentSession extends ChangeNotifier`
  (no Riverpod). Loads bytes via SAF, decodes (never throws), holds the `re_editor`
  controllers for the source, tracks a JSON **view mode** (`pretty` / `tree` / `raw` /
  `minified` / `edit`), parses to a `JsonNode` tree + validity + stats + NDJSON detection +
  a **lenient-read notice** ("read leniently; will save as strict JSON"), drives drafts +
  autosave, and saves via `AtomicSaver` + `SafSaveTarget` **with `JsonWellFormedGate`**,
  preserving encoding/line endings and using a **chosen indentation**. Builds metadata with
  `formatFields` (top-level type, item count, key count, depth). Mirrors `MdDocumentSession`.
- `lib/formats/json/json_well_formed_gate.dart` — `JsonWellFormedGate implements SaveGate`
  (§3.4): parses strict, returns `GateResult.invalid(message-with-line)` on failure.
- `lib/formats/json/json_session_manager.dart` — `JsonSessionManager` +
  `jsonSessionManagerProvider` (map keyed by `tab.id`; `sessionFor`/`peek`/`release`/
  `retainOnly`). Mirrors the MD manager.
- `lib/formats/json/json_document_view.dart` — the tab body: loading spinner / friendly
  failure view (Retry) / ready view; ready view shows the lenient-read banner + draft banner
  + the pretty/tree/raw/minified/edit view for the current mode.
- `lib/formats/json/json_toolbar.dart` — action bar: view toggles (pretty/tree/raw/minified/
  edit), undo/redo (edit), find, format / minify, validate, expand/collapse all (tree),
  save, read-aloud, and an overflow menu (find & replace, JSONPath, insights, file info,
  diff, split, merge, copy full/minified, share, share-as-zip, print, export). Mirrors
  `MdToolbar`.

### New — output helpers (mirror MD/TXT, reuse Phase 5 services)

- `lib/core/export/json_exporter.dart` — `JsonExporter implements FormatExporter`,
  `formatId => 'json'`, targets `{csv, yaml, pdf, html, plainText}`. CSV via the flattener
  (§3.6), YAML via `json_yaml.dart`, PDF/HTML via the shared writers over pretty text,
  plainText passes the source through. Unsupported target → `UnsupportedExportException`.
- `lib/formats/json/json_output_actions.dart` — `JsonOutputActions` (`formatId = 'json'`):
  `shareFile`, `shareAsZip`, `printDoc`, `runExport`, `saveExport`, `shareExport`. Mirrors
  `MdOutputActions`.
- `lib/formats/json/json_export_sheet.dart` — target picker → run → share/save.
- `lib/formats/json/json_info_sheet.dart` — stats + metadata + insights rows.
- `lib/formats/json/json_save_options_sheet.dart` — indentation (2 / 4 / tab / minified) +
  encoding + line-ending choose → save.
- `lib/formats/json/json_split_merge_actions.dart` — split dialog → per-part
  `saf.createDocument`; merge picks JSON files and concatenates top-level arrays.
- `lib/formats/json/json_diff_sheet.dart` — pick a second JSON file via SAF → show the diff
  (added / removed / changed) from `JsonDiff`.

### Changed — wiring

## 6. How we honour the project rules

- **Open source only** — no new package; `dart:convert` + our own reader/renderer; no
  Syncfusion (CLAUDE.md §3.1).
- **Never crash on bad input** — the parser never throws; corrupt/empty/wrong-encoding lands
  in a friendly `failed`/empty state; invalid JSONPath/schema degrade to a message
  (CLAUDE.md §3.4).
- **Scoped storage only** — open/save/diff/merge all go through SAF + `createDocument`; no
  broad storage (CLAUDE.md §3.3).
- **Atomic saves + preserve encoding/line endings** — reuse `AtomicSaver` + `SafSaveTarget`
  exactly as MD/TXT, now with the JSON well-formedness gate (CLAUDE.md §3.5).
- **Never lose edits silently** — reuse drafts/autosave + the shell's unsaved-changes close
  guard; the gate blocks a broken overwrite but offers save-as-a-copy (CLAUDE.md §3.6).
- **Offline-first** — no remote fetch; exports are self-contained (CLAUDE.md §3.2).
- **Simple English** — all UI strings, the change log, and comments in plain English.

## 8. Out of scope for this phase (noted, not dropped)

- **Full JSON5 / full JSON Schema** — we support a pragmatic subset (§3.1, §3.5); the limits
  are documented and unknown constructs degrade gracefully.
- **Advanced structural tree edits** (drag-reorder, move a subtree) — first cut is
  edit-value / edit-key / add-child / delete (§3.3).
- **Large-file streaming** for huge JSON — handled cross-cutting in **Phase 10**.
- **Syntax colouring inside the raw `re_editor` surface** — the *pretty* view is coloured;
  the raw editor is plain monospace (a later polish item, matching MD).
