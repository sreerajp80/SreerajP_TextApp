# Settings: one card per section, tap to open its own page

**Status:** completed

## What the issue is

Right now the Settings screen is one long scrolling list. All seven groups
(Appearance, Editor, Files & Tabs, Speech, Sync, Security, About) are stacked
on top of each other with `Divider()`s in between (see
[settings_screen.dart](../lib/shell/settings/settings_screen.dart)).

The request: make each section a **card**. Tapping a card should **open a
separate page** that shows only that section's settings.

So Settings becomes a short menu of cards. Each card leads to its own detail
page.

## The plan for the fix

Keep every section widget's real content exactly as it is today. Only change
how the sections are shown:

1. **A menu of cards.** Rewrite `SettingsScreen` so its body is a list of
   `Card`s — one per section. Each card is a `ListTile` inside a `Card` with:
   - a leading icon (a fitting Material icon per section),
   - the section title (reuse the existing `*SectionTitle` l10n strings),
   - a one-line description (new l10n strings, see step 4),
   - a trailing chevron (`Icons.chevron_right`).

   Tapping a card pushes a `MaterialPageRoute` to the section's detail page.

2. **A generic detail page.** Add a new file
   `lib/shell/settings/settings_detail_screen.dart` with a small
   `SettingsDetailScreen` widget: a `Scaffold` with an `AppBar` (title = the
   section title) and a scrolling body holding the one section widget passed in.

3. **Hide the in-body header on the detail page.** Each section widget starts
   with a `SettingsSectionHeader` (the section title in the primary color). On a
   dedicated page the `AppBar` already shows that title, so the in-body header
   would be redundant. Add an optional `bool showHeader` (default `true`) to
   each of the seven section widgets and wrap their `SettingsSectionHeader` in
   `if (showHeader)`. The detail page passes `showHeader: false`. Default `true`
   keeps every other caller and existing widget test unchanged.

4. **New l10n strings.** Add one short description string per section to
   `lib/l10n/app_en.arb` (e.g. `appearCardSubtitle`,
   `editorCardSubtitle`, `filesTabsCardSubtitle`, `speechCardSubtitle`,
   `syncCardSubtitle`, `securityCardSubtitle`, `aboutCardSubtitle`), each with an
   `@`-description entry, then regenerate the localizations with
   `flutter gen-l10n` (config is in [l10n.yaml](../l10n.yaml)).

5. **Update the screen test.** `test/shell/settings/settings_screen_test.dart`
   currently pumps `SettingsScreen` and reads section content directly (About
   values, the confirm-before-overwrite switch). After this change that content
   lives behind a card, so those tests must first **tap the card** to open the
   detail page, then assert. The "all seven section headers render" test still
   passes because the card titles reuse the same strings, but I will rename/adjust
   it to reflect that these are now cards. Add one new test: tapping a card opens
   its detail page.

## Notes / choices

- No section's actual settings, controllers, or behavior change — only the
  layout and navigation.
- Existing widget tests that pump a single section directly
  (`security_settings_test.dart`, `tabs_settings_test.dart`, etc.) are
  unaffected because `showHeader` defaults to `true`.
- Icons planned: Appearance `Icons.palette_outlined`, Editor `Icons.edit_outlined`,
  Files & Tabs `Icons.tab_outlined`, Speech `Icons.record_voice_over_outlined`,
  Sync `Icons.sync`, Security `Icons.lock_outline`, About `Icons.info_outline`.
