# Plan — Phase 12: P2P LAN sync

**Status:** completed

Plan for **Phase 12** in [implementation-plan.md](../docs/implementation-plan.md)
(tasks 12.1–12.8) and the design in [architecture.md](../docs/architecture.md) §9.
**Security-sensitive.** Follows [security-rules.md](../docs/security-rules.md) throughout.

Phase 12 depends only on Phase 1 (data layer + secure storage), which is done. It is a
parallel track and does **not** need the format phases.

---

## 1. What the issue is

There is no way to move app data between two devices. The `lib/sync/` folder has only a
`.gitkeep`. We must build offline, same-LAN, device-to-device sync with:

- **no server, no account, no internet** — plain local TCP sockets only,
- **out-of-band pairing** — a fresh high-entropy code per session, shown as QR + text,
  scanned or typed; the code **never** goes on the wire,
- **payload-layer crypto** — PBKDF2-HMAC-SHA256 key from the code, every wire message
  sealed with AES-256-GCM; a wrong code fails the GCM tag and aborts,
- **hostile-peer hardening** — bounded reads (separate handshake vs payload caps),
  timeouts, payload caps, single client at a time, strict QR parsing,
- **add-only, client-wins merge** — the receiver never has its data overwritten; only an
  allow-list of non-sensitive settings may sync; security/identity state never syncs.

All of the sync engine (12.1–12.7) is pure Dart / Riverpod and is fully testable here over
**loopback** (no devices). Only the QR camera scan and a real two-device transfer need a
device; those are recorded as **owed manual checks** (the two-device run is Phase 14.2).

## 2. Key design decisions (please confirm at approval)

### 2.1 What data is syncable (the categories)

The transport is app-agnostic (moves opaque strings). At the payload layer we sync these
**categories**, each with a natural key for add-only merge:

| Category | Source (Phase 1) | Natural key | Notes |
|---|---|---|---|
| `favorites` | `FavoritesRepository` | `fingerprint` | Content-keyed, portable across devices. |
| `bookmarks` | `BookmarksRepository` | `fingerprint` + `position` + `label` | Content-keyed; `id` is dropped on import (re-assigned locally). |
| `recents` | `RecentsRepository` | `fingerprint` | The SAF `uri` is device-local, so a synced recent may be **unavailable** on the other device — this is fine (Phase 2.3 already shows stale/unavailable recents with a remove option). |
| `settings` | `KeyValueStore` (prefs) | the setting key | **Allow-list only** of non-sensitive keys (theme, font, line spacing, word-wrap, tab prefs, sync tunables). Never sensitive keys. |

**Never synced (hard rule):** `device_key`, `app_lock_pin`, any secure-storage/identity
value. These are excluded structurally (see 2.3) and by an allow-list, not by a filter that
could be forgotten.

Recents carry only `fingerprint`, `displayName`, `mimeType`, `size`, `lastOpenedAt` — the
**`uri` is included** so a re-pick can match, but the receiver treats a synced recent as
"file identity known, may be unavailable" until the user re-picks it. (Alternative: drop
`uri` entirely and only keep the name/fingerprint — I recommend keeping it as an optional
hint; it is not a secret.)

If you prefer a **smaller first cut** (e.g. only `favorites` + `bookmarks` + `settings`,
dropping `recents`), say so at approval and I will trim it. The category set is centralized
so adding/removing one is a one-line change.

### 2.2 Payload shape (wire JSON, one line, AES-GCM sealed)

```json
{
  "app": "text_data",
  "payloadVersion": 1,
  "syncMode": "full",
  "records": {
    "favorites": [ { "fingerprint": "...", "displayName": "...", "addedAt": 0 } ],
    "bookmarks": [ { "fingerprint": "...", "label": "...", "position": 0, "createdAt": 0 } ],
    "recents":   [ { "fingerprint": "...", "uri": "...", "displayName": "...", "size": 0, "lastOpenedAt": 0 } ]
  },
  "settings": { "theme_mode": "dark", "font_scale": 1.0 }
}
```

- A **category key is present only if it is part of this sync** (so the receiver tells
  "0 sent" from "not included") — matches arch §9.4.
- `syncMode` is `full` (fresh device: apply everything incl. settings) or `incremental`
  (add-only: skip records whose natural key already exists; settings apply **fill-only**).

### 2.3 Secret lifecycle (task 12.6)

Today **no synced category holds a per-record secret** (favorites/bookmarks/recents/settings
are all non-secret). But the security rules require the machinery so future secret-bearing
records are safe. I will:

- Build a small `SecretResealer` in `sync_crypto`/`payload`: host side
  `deviceKey-decrypt → sessionKey-encrypt`; client side `sessionKey-decrypt →
  deviceKey-encrypt`. Pure, unit-tested for a full round-trip (device-key → session-key →
  device-key), proving nothing secret is ever stored under the session key at rest.
- Provision a **device key** on first use with `Random.secure()`, stored **only** under the
  existing sensitive key `device_key` in secure storage (already in
  `KeyValueStore.defaultSensitiveKeys`). It is never synced and never logged.
- The current payload builder marks that it carries **no** secret records, so the resealer
  is exercised by tests but not on any live category yet. This keeps the rule satisfied and
  the door open, without inventing fake secret data.

### 2.4 Scope: engine now, UI included, camera scan owed

I will build **all eight tasks**, including the Sync UI (12.8) with `qr_flutter` (render)
and `mobile_scanner` (scan) plus a manual IP/port/code form. The camera scan path and the
real two-device transfer can only be verified on a device, so they are recorded as **owed
manual checks** (two-device is formally Phase 14.2). Everything else is covered by unit /
loopback / widget tests that run here.

## 4. The plan for the fix (task by task)

All new files live under `lib/sync/`. Lower layers stay pure Dart so they test without a
device. Nothing logs payloads, secrets, keys, or the code — enforced by review and a test
that greps for accidental logging.

### 4.1 Sync constants — `lib/sync/sync_constants.dart` (task 12.1)
Central home for every tunable and wire literal so host and client stay in lockstep and the
caps are auditable in one place:
- pairing alphabet (no `0 O 1 I L`), code length (→ ~320 bits), group/format size,
- PBKDF2 iterations, salt size, GCM nonce size (12), key size (32),
- handshake strings (`HELLO_SYNC`, `ACCEPT_SYNC`, `DENIED`),
- bounded-reader caps (handshake line cap vs payload line cap), payload caps (max records
  per category, max per-field length),
- timeouts (`connectTimeout`, `socketTimeout` per line, long `payloadWaitTimeout`),
- QR URI scheme / host / version, default port range,
- `syncMode` keys, category keys, settings allow-list.
- Referenced by the crypto/transport/payload tests.

### 4.2 Crypto — `lib/sync/sync_crypto.dart` (task 12.2)
- `randomBytes(n)` from `Random.secure()`.
- Pairing code `generate` / `normalize` (upper-case, strip spaces, map look-alikes) /
  `format` (grouped for reading). The code never goes on the wire.
- `deriveKey(code, salt)` — PBKDF2-HMAC-SHA256 (`pointycastle`), constants from 4.1.
- `encryptWire(key, plaintext)` / `decryptWire(key, wire)` where
  wire = `base64(nonce(12) || ciphertext+tag)`, one line, AES-256-GCM (`encrypt`). Wrong key
  → GCM tag fails → throws.
- Strict QR URI `build` / `parse` (checks scheme/host/version/port/code; rejects foreign or
  malformed URIs — never throws to the UI, returns a typed failure).
- `SecretResealer` helpers (2.3).
- Tests: wire round-trip; wrong key throws; QR parse rejects malformed/foreign; code
  normalize; resealer round-trip.

### 4.3 Bounded line reader — `lib/sync/bounded_line_reader.dart` (task 12.3)
- Wraps a byte stream, buffers, splits on `\n`, **caps line length** (separate handshake vs
  payload caps), exposes a `closed` future. No unbounded `readLine` (guards slow-loris /
  giant-line DoS). Over-cap line → typed error and the connection is dropped.
- Tests: over-long line rejected at the cap; a stream drop resolves `closed`.

### 4.4 Transport — `lib/sync/sync_transport.dart` (task 12.4)
- `SyncHost` (`ServerSocket`) and `SyncClient` (`Socket`), **connect-then-choose** flow
  (arch §9.3): host sends `base64(salt)` in clear → client sends `encrypt(HELLO_SYNC)` →
  host verifies and sends `encrypt(ACCEPT_SYNC)` immediately (both show "connected") → host
  holds open → on the sender's action host sends `encrypt(payloadJson)`.
- Wrong code: host's HELLO decrypt throws → host replies `DENIED` and keeps listening; or
  client's ACCEPT decrypt throws → client reports "incorrect code".
- Single client at a time; connect + per-line socket timeouts; long `payloadWaitTimeout` for
  the payload read. `sendToConnectedClient` with no client throws.
- App-agnostic: moves opaque strings only (no app types).
- Tests (loopback): happy path connects + receives the pushed payload; wrong code rejected
  and never connects; `sendToConnectedClient` with no client throws.

### 4.5 Payload build / validate / merge — `lib/sync/payload.dart` (task 12.5) + secret lifecycle (12.6)
- `SyncPayload` model + `build(categories, syncMode)` from the chosen repositories/settings.
- `validateAndParse(String json)` — treats every payload as **hostile**: parse, enforce
  caps (max records per category, per-field length), reject malformed/oversized **before**
  it touches the DB; returns a typed result or a friendly error.
- `merge(payload, syncMode)` — add-only, client-wins:
  - `full`: apply all records + all settings.
  - `incremental`: skip records whose natural key already exists; settings **fill-only**
    (only if the receiver has not set that key).
  - Returns per-category counts (added / kept) + settings (applied / kept) for the summary.
- Category abstraction: a small registry mapping a category key → (export records,
  natural-key fn, import/merge). Adding a category is one entry.
- Secret lifecycle machinery (2.3) lives here + `sync_provider`.
- Tests: validate rejects oversized/malformed and caps counts; merge skips duplicates and
  applies settings fill-only; secret round-trip; no-secret-in-logs grep test.

### 4.6 Sync provider — `lib/sync/sync_provider.dart` (task 12.7)
- Riverpod notifier holding sync state (idle / hosting-waiting / connected / sending /
  done / error on the host; scanning / connecting / waiting / applied / error on the
  client). Runs the transport, builds the payload from chosen categories, applies imports
  via the repositories, provisions/reads the device key.
- Test: full loopback flow through the provider (host connect → choose → push → client
  import summary), using in-memory FFI DB + loopback sockets.

### 4.7 Sync UI — `lib/sync/` widgets + `lib/shell/settings/` sync entry (task 12.8)
- **Host — two tabs:** (1) connection details — QR (`qr_flutter`) + IP / port / code as
  selectable text with copy, a live status chip ("Waiting…" → "Connected ✓"), Stop; (2)
  choose what to share — **Full Sync** action + per-category checkboxes with the note
  *"This won't override anything already on the other device; on a conflict the other device
  keeps its data."* Actions disabled until connected.
- **Client:** scan-QR button (`mobile_scanner`) + manual IP/port/code form → waiting screen
  → summary (added / kept / applied).
- **Suppress idle/background auto-lock while the sync screen is open** (scoped to that
  screen); optional screenshot protection (`FLAG_SECURE`) while the code/QR shows — the full
  app-lock + FLAG_SECURE toggles are Phase 11.6 / 13.2, so here I add the hook and a safe
  default.
- An entry point to reach Sync (a shell action / settings row). The full Settings **Sync
  section** (11.5) and **Security section** (11.6) are Phase 11; here I add the minimal
  entry + the syncable-categories default.
- Tests (widget): status chip updates on connect; actions gated until connected; summary
  renders counts. (Camera scan is an owed manual/device check.)
