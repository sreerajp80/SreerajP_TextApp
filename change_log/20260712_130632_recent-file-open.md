# Fix recent files not opening in the editor

Implemented
[plans/20260712_130404_recent-file-open.md](../plans/20260712_130404_recent-file-open.md).

## What changed

- Updated `OpenFileAction` to capture the SAF service and the recents, tabs,
  and shell destination controllers before asynchronous file work begins.
- The open flow now keeps using those valid controller references after
  `recordOpen()` refreshes and disposes the Recent files list widget.
- Kept the existing scoped-storage read, content fingerprint, recent-entry
  update, tab-cap handling, and Editor navigation behavior.
- Added a widget regression test that forces the recent list to enter loading
  state during a tap and verifies that a tab is still opened and the selected
  shell destination becomes Editor.

## Verification

- `dart format lib/shell/open_file_action.dart test/support/test_support.dart test/shell/home_screen_widget_test.dart` — passed.
- `flutter test test/shell/home_screen_widget_test.dart` — passed, 5 tests.
- `flutter analyze` — passed with no issues.
- `flutter test test/shell` — the new regression test passed. The suite reached
  61 tests, with one unrelated existing failure in
  `test/shell/theme_switch_widget_test.dart`: it tries to tap a `Dark` control
  that is not present on the current first-level Settings card screen. Running
  that test alone gives the same failure.

## Security and data safety

- File access still uses the existing SAF service and persisted content URI.
- No file content is logged or sent anywhere.
- Save behavior and unsaved-edit protection are unchanged.
