## Stage A — Security hardening + production keystore (task 13.1)

- **Audit:** wrote `docs/security-audit-phase13.md` recording each rule from
  `docs/security-rules.md` with evidence. The code was already clean (one guarded,
  secret-free `debugPrint`; `Random.secure()` confirmed in sync crypto).
- **Rule-guard tests** (`test/security/`):
  - `logging_audit_test.dart` — scans all of `lib/` for logging calls; only a short,
    reviewed allow-list (the guarded config `debugPrint`) is permitted, so any new
    `print`/`debugPrint`/`developer.log` fails CI.
  - `random_secure_test.dart` — locks `Random.secure()` in `sync_crypto.dart` (no bare
    `Random(`), the look-alike-free alphabet, and ~320-bit code entropy.
  - Payload caps / bounded-reader coverage stays in the existing `test/sync/` suites.
- **Keystore wiring (no secrets in repo, none generated by the AI):**
  - `android/app/build.gradle.kts` — added a `release` `signingConfig` that loads
    `../key.properties` **only if it exists** (debug fallback otherwise).
  - `android/key.properties.example` — committed template (placeholders only).
  - `android/.gitignore` — confirmed `key.properties`, `*.jks`, `*.keystore` ignored.
  - `docs/release-signing.md` — the `keytool` command + steps to build a signed release.
    The developer generates the `.jks` and fills `key.properties` themselves.

## Stage B — App-lock (PIN + biometric + recovery code) + screenshot protection (task 13.2)

Re-scoped after user feedback to add biometric unlock and a recovery code.

- **Package:** `local_auth` (BSD-3). `MainActivity` now extends `FlutterFragmentActivity`;
  `USE_BIOMETRIC` permission added.
- **Secrets model (never synced, never logged):** PIN and recovery code are stored only as
  **salted PBKDF2-HMAC-SHA256 hashes** under the Keystore-backed keys `app_lock_pin` /
  `app_lock_recovery` (both added to `KeyValueStore.defaultSensitiveKeys` and
  `SyncConstants.neverSyncKeys`). The recovery code is shown once at setup; entering it
  unlocks and **forces a new PIN** (and regenerates a fresh recovery code).
- **New `lib/core/security/`:** `app_lock_hasher.dart`, `app_lock_repository.dart`,
  `biometric_service.dart`, `window_security.dart`, `app_lock_controller.dart`,
  `app_lock_gate.dart`, `security_providers.dart`, and the `lock_screen` / `set_pin_screen`
  / `recovery_code_screen` surfaces.
- **New Kotlin `WindowSecurityChannel.kt`** (method channel `app/window_security`) toggles
  `FLAG_SECURE`; driven app-wide when screenshot protection is on, and forced on the sync
  code/QR screen.
- `lib/app.dart` wraps `home:` with the lock gate; `security_section.dart` makes the toggles
  real (set PIN → show recovery, change PIN, regenerate recovery, biometric toggle).
- **Tests:** `test/core/security/` — hasher round-trip, controller flows (PIN / biometric /
  recovery → force-new-PIN), and the gate widget test.

## Stage D — Localization, full extraction (task 13.4)

- **Infrastructure:** added `flutter_localizations` + `intl`, `flutter: generate: true`,
  `l10n.yaml`, `lib/l10n/app_en.arb`, delegates + `supportedLocales` in `app.dart`, and a
  shared `localizedApp(...)` test helper.
- **Every user-facing string extracted** into the ARB and replaced with
  `AppLocalizations.of(context)` across: app title, home, onboarding, app shell, the
  security/lock/PIN/recovery flow, **all settings sections**, **all sync screens**,
  **tabs + dialogs** (unsaved-changes, overwrite, degraded/large-file, placeholder), and
  **all five formats** — TXT, Markdown, CSV, JSON, XML — including toolbars, tree views,
  find/replace panels, info/encoding/export/save/split-merge sheets, output actions,
  read-aloud buttons, link warnings, and every failure/empty state.
- Reusable shared key families (`action*`, `save*`, `out*`, `find*`, `export*`, `split*`,
  `readAloud*`, `info*`, `sync*`, `tab*`, `draft`, `fail`) keep the ARB DRY across formats.
- **Tests:** widget tests that pump localized screens were updated to supply the delegates
  (via `localizedApp`); added `test/l10n/localization_test.dart` (delegate resolves, plus
  placeholder + plural checks).

## Stage E — Error / empty-state polish (task 13.5)

- **Non-blocking snackbars app-wide:** added a global `SnackBarThemeData`
  (`behavior: floating`, `showCloseIcon: true`) in `AppThemes._build`, so every notice floats
  and never blocks content — no per-call-site edits.
- Confirmed each format's bad-file path shows a consistent, friendly, never-crashing failure
  view (error icon + title + message + Retry) and friendly empty states (all localized and
  reviewed in Stage D); the parser failure-path suites stay green.
- **Test:** `test/shell/error_polish_test.dart` (light/dark/sepia snackbar behavior).
- Scope note: kept the per-format failure/empty widgets rather than extracting shared ones —
  they are already consistent and localized, so a refactor would be churn with no behavior
  gain.
